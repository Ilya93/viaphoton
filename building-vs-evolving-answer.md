# Building vs. Evolving

I use roughly the same stack in both cases, but I'm optimizing for different things. When I'm building from zero, I'm optimizing for speed of learning. I want something in front of real users fast, and I'm willing to throw half of it away. When I'm working on a product that already has customers, I'm optimizing for not breaking them. Every change has to be reversible, observable, and safe to roll back. Same tools, different defaults.

## Building from scratch

I start with the user, not the stack. A few real conversations, a one-page problem statement, and a short list of the three or four risks that could actually kill the project. Then I draft a walking-skeleton architecture and ship a thin vertical slice in the first iteration: auth, one real entity, one API, one page in production. Every later decision gets validated against a running system instead of a diagram. I run iterations with a real demo to the people who asked for the thing.

On process, I keep things lightweight. A short async standup in writing, a working demo whenever there's something worth showing, and a quick retro when we hit a milestone or something broke. I don't believe in scrum ceremony for its own sake. What I care about is a tight feedback loop with whoever asked for the work, and anything expensive to reverse gets written down first: a short doc with the trade-off, read by someone other than the author before we commit to it.

On tools, TypeScript everywhere, because shared types across frontend, backend, and queue payloads catch a lot of bugs before they ship. Next.js and React for the UI. NestJS on Node.js when the domain is modular. FastAPI in Python where the work is genuinely Python: OCR, NER, embeddings. MongoDB when the domain is naturally document-shaped. I've picked it before in cases where every record had a different shape, and a relational model would have meant either an EAV mess or a schema migration every time the product added a new category. When I need strict multi-document consistency, like billing or scoring, I reach for Postgres instead. Redis for cache, sessions, and rate limiting. Bull or BullMQ for in-process background work. SQS plus Lambda plus SES when I need elastic fan-out, for example big email batches where I want the queue to scale on its own, while lighter in-app jobs stay on Bull. Stripe for billing, with webhook idempotency keyed through Redis so a retried payment event doesn't double-charge. Cognito or Better Auth for identity. Ably for real-time, so I don't own sticky-session WebSockets. Turborepo with pnpm for monorepos, Docker, GitHub Actions, Sentry, Pino for structured logs, Figma, and preview deploys on every PR.

A few basics I set up early so the team doesn't pay for them later: TypeScript in strict mode, lint and CI on every PR, PR reviews between pairs, and structured logs with a request ID from the first commit. I don't chase coverage numbers. I write tests where they actually catch things: unit tests around tricky logic, integration tests where we touch the DB or a third-party, and a handful of end-to-end tests on flows a user genuinely can't lose.

## Evolving a product already on the market

I saw a very different side of this when I on enterprise e-commerce platform. Millions of active merchants, some of them running their stores since the mid-2010s. You can't just rewrite things in that environment. Every decision is weighed against "what happens to the merchants using this today," and that changes how you work.

The clearest example was the checkout. Two versions lived side by side: a legacy one in Elm, and a newer one in Vue. When I built the shipping discounts feature, it had to work in both, with the same logic and a shared CSS architecture, because you can't force every merchant onto the new checkout overnight. That's strangler-fig in practice: you don't rewrite the old path, you put the new one next to it, keep them behaving the same, and let traffic move over time. The real lesson for me was that a feature PR on a mature product is rarely pure feature work. It's feature plus compatibility plus a migration story, and if you don't plan for the last two, you ship bugs.

The second thing was how much code review matters in that kind of codebase. Every PR was read carefully, often by more than one person, often with follow-ups that sent you back to rework. It slows you down at first. Then you see it's the main thing keeping a codebase that size coherent across many teams. I kept the habit: I write the "why" in the PR description so the reviewer doesn't have to guess my intent, and I don't rubber-stamp other people's PRs either.

The third lesson was about leverage. Some of the most useful work on a mature product isn't a feature at all. I spent real time on internal npm packages other teams depended on, and on build-pipeline tuning. A thirty-second improvement to the local feedback loop, multiplied across a few hundred engineers, pays for itself in a week. On a small greenfield you don't notice it. On a big product, it's quietly one of the most valuable things you can do.

The fourth thing is observability, and I actually learned that one on my own product. We had chat messages dropping now and then, and `console.log` was useless. Once I added structured logs with a request ID, a user ID, and an event ID on every line, I found the cause (an upstream rate-limit) in minutes. Same pattern later: a slow memory leak I only caught with a heap snapshot, and a slow query that turned out to be a missing compound index, obvious once I ran `explain()`. The point isn't "add logging after an incident." Observability is a day-one tool.

A few rules I took with me from that period: migrations are online and backwards-compatible or they don't ship. User-visible changes go behind a flag and roll out gradually. Refactors don't ride inside feature PRs, because they make the diff hard to review. Dependency upgrades get their own PRs, so when something breaks you know exactly what caused it. And when something does break, there's a short, blameless write-up with real follow-up actions, or it didn't really happen.
